## Bugs / definite issues

### 1. Tautological assertion on `unsigned` field — `scanner.c:128`

```c
assert(scanner->length >= 0);
```

`length` is declared `unsigned` in `scanner.h:24`, so this is always true and will trip `-Wtype-limits` on a strict build. The follow-up `assert(scanner->back(scanner) >= 0)` is the meaningful invariant; this one can be deleted.

### 2. Unreachable branch — `parser.c:1022-1024` in `parse_quoted_literal_block`

```c
if (indent != current_indent || scanner->lookahead != adornment) {
  break;
}
if (scanner->lookahead != adornment) {     // dead — covered by the OR above
  return parse_text(scanner, false);
}
```

The second test can never fire. Either the second branch was meant to live before the first (the comment "indentation was broken" reads like it was), or it's leftover from a refactor. Either delete it or restructure so the "fall back to text" path is actually reachable when the literal block dies on a non-adornment line.

### 3. Standing TODO in correction-mode fallback — `scanner.c:160-167`

The author's own note: when `T_INVALID_TOKEN` is in the valid set (tree-sitter error correction), the scanner falls back to emitting text. This couples scanner behavior to the JS grammar in a fragile way and should ideally be moved to JS as the comment says. Worth tracking; the current behavior is "works but not where it belongs".

## Architectural / structural concerns

### 4. The dispatcher's load-bearing invariant is implicit

`rst_scanner_scan` is a long if/else chain where most branches call `scanner->advance` before deciding whether to commit. Tree-sitter does **not** roll back lexer state when a scanner returns false — every advance persists. `table.c` calls this out at length, but `scanner.c` and `parser.c` do not. The dispatch order in `rst_scanner_scan` (lines 169-309) is therefore safety-critical — e.g. `parse_overline` must run before the `+`/`=` table dispatch, the inline-markup branch must come after `_`-target/`|`-substitution, etc.

Recommendation: a one-paragraph header on `rst_scanner_scan` stating "any branch reached here owns the lexer state for the rest of this scan; reordering branches is not safe; sub-parsers that advance must commit to a token (true) or carefully restore via mark_end (false but boundary already pinned)" would save future readers a lot of reverse-engineering.

### 5. Indent-stack silent failures — `scanner.c:68-90`

- `push` silently no-ops on overflow (capacity 99).
- `pop` silently returns 0 on underflow.
- `serialize` truncates on buffer overflow.

The capacity is plausibly fine for real docs, but the failure mode (silent state corruption that surfaces as confusing parse trees) is hard to diagnose. At least an `assert(false && "indent stack overflow")` in a debug build would surface the problem during testing. The serialize truncation is more concerning — across an incremental edit, a deserialize on the truncated buffer leaves the stack shorter than it was before serialize, which can mis-match indent levels in the rest of the document.

### 6. `is_start_char` / `is_end_char` are linear scans on a hot path — `chars.c:127-150`

Both delegate to `is_delim_char`, which iterates `delim_chars` (~150 entries) and `delim_chars_range` (~48 ranges) linearly on every character. These are called from the inline-markup loop (`parse_inner_inline_markup`) on essentially every character in inline content. Not a correctness issue, but a real perf concern.

A sketch fix is at the bottom of this document.

### 7. `conflicts: $ => []` next to overlapping `_directive_body` — `grammar.js:103, 625-657`

Four overlapping alternatives:

```js
seq(args?, opts?, blankline, content),            // with content
seq(choice(args, opts, seq(args, opts)), dedent), // without content
seq(newline, choice(...)),                        // content on first line
seq(args, content),                               // multiline args
```

Tree-sitter would have failed generation if there were unresolved conflicts, so today this works. But every future tweak to directive parsing is a roll-of-the-dice on whether tree-sitter still resolves it without an explicit `conflicts` entry. Worth either (a) writing the actually-needed conflicts down so future readers know what tree-sitter is auto-resolving, or (b) refactoring into fewer non-overlapping alternatives.

### 8. Hardcoded magic numbers

- `TAB_STOP = 8` (`chars.h:20`) — no way to change at parse time.
- `MAX_SCHEMA_LEN = 20` (`parser.c:1536`) — silent truncation of long URI schemes.
- `RST_SCANNER_STACK_MAX_CAPACITY = 99` — odd choice; powers-of-two are more conventional.

None of these are urgent, but `MAX_SCHEMA_LEN` truncating silently means an unusually-long valid scheme gets misclassified rather than rejected loudly.

# Sketch fix for #6 — fast `is_start_char` / `is_end_char` / `is_delim_char`

## Problem

Each of `is_start_char(c)` and `is_end_char(c)` does:

```c
for (int i = 0; i < length; i++) if (c == arr[i]) return true;
// ... and then ...
return is_delim_char(c);  // which itself does two more linear scans
```

For a typical ASCII character that is *not* a delimiter (the common case in inline markup scanning), each call iterates ~96 + ~150 + ~48 entries before returning false.

## Approach: two-tier lookup

Most input characters in real-world RST are ASCII. A 128-bit ASCII bitmap covers them in O(1) with one shift + one AND. Non-ASCII (BMP and beyond) goes through a sorted-array binary search and a sorted-range binary search — O(log n).

### Generated layout (replaces current `punctuation_chars.h`)

`gen_punctuation_chars.py` would emit, per character set:

```c
// One 128-bit bitmap covering ASCII (codepoints 0x00..0x7F).
const uint32_t start_chars_ascii_bitmap[4] = { 0x..., 0x..., 0x..., 0x... };

// Non-ASCII codepoints, sorted, deduped.
const int32_t start_chars_unicode[] = { 0xab, 0xbb, 0x2018, /* ... */ };
const size_t  start_chars_unicode_len = /* ... */;
```

Same shape for `end_chars` and `delim_chars`. The range table:

```c
// Sorted by lo bound, non-overlapping.
const int32_t delim_chars_range[][2] = { /* sorted */ };
const size_t  delim_chars_range_len = /* ... */;
```

(Range entries that fall entirely below 0x80 can be folded into the ASCII bitmap; today there don't appear to be any, but the generator should handle it.)

### Lookup primitives in `chars.c`

```c
static inline bool ascii_bitmap_test(const uint32_t bitmap[4], int32_t c) {
  // Caller must have already checked c >= 0 && c < 128.
  return (bitmap[(unsigned)c >> 5] >> ((unsigned)c & 31)) & 1u;
}

static bool sorted_array_contains(const int32_t* arr, size_t n, int32_t c) {
  size_t lo = 0, hi = n;
  while (lo < hi) {
    size_t mid = lo + (hi - lo) / 2;
    if (arr[mid] == c) return true;
    if (arr[mid] < c)  lo = mid + 1;
    else               hi = mid;
  }
  return false;
}

static bool sorted_range_contains(const int32_t (*ranges)[2], size_t n, int32_t c) {
  // Find rightmost range with lo <= c; then check c <= hi.
  size_t lo = 0, hi = n;
  while (lo < hi) {
    size_t mid = lo + (hi - lo) / 2;
    if (ranges[mid][0] <= c) lo = mid + 1;
    else                     hi = mid;
  }
  if (lo == 0) return false;
  return c <= ranges[lo - 1][1];
}
```

### Replacement classifiers

```c
static bool is_delim_char(int32_t c) {
  if ((uint32_t)c < 128)
    return ascii_bitmap_test(delim_chars_ascii_bitmap, c);
  return sorted_array_contains(delim_chars_unicode, delim_chars_unicode_len, c)
      || sorted_range_contains(delim_chars_range, delim_chars_range_len, c);
}

static bool is_start_char(int32_t c) {
  if ((uint32_t)c < 128) {
    return ascii_bitmap_test(start_chars_ascii_bitmap, c)
        || ascii_bitmap_test(delim_chars_ascii_bitmap, c);
  }
  return sorted_array_contains(start_chars_unicode, start_chars_unicode_len, c)
      || sorted_array_contains(delim_chars_unicode, delim_chars_unicode_len, c)
      || sorted_range_contains(delim_chars_range, delim_chars_range_len, c);
}

static bool is_end_char(int32_t c) {
  if ((uint32_t)c < 128) {
    return ascii_bitmap_test(end_chars_ascii_bitmap, c)
        || ascii_bitmap_test(delim_chars_ascii_bitmap, c);
  }
  return sorted_array_contains(end_chars_unicode, end_chars_unicode_len, c)
      || sorted_array_contains(delim_chars_unicode, delim_chars_unicode_len, c)
      || sorted_range_contains(delim_chars_range, delim_chars_range_len, c);
}
```

## Cost / risk

- **Code surface:** `chars.c` gains ~30 lines of helpers and shrinks `is_start_char` / `is_end_char` / `is_delim_char` to a few lines each. `gen_punctuation_chars.py` grows by ~40 lines (split ASCII vs. non-ASCII, dedupe, sort, sort the range table by lo bound, emit bitmap as four hex constants).
- **Output is still a single header** (`punctuation_chars.h`), so the build graph doesn't change.
- **Behavior must match exactly.** Add a one-shot unit-test step in the generator that walks 0x00..0x10FFFF, evaluates the old linear-scan logic and the new bitmap-and-bsearch logic, and asserts equivalence. Cheap insurance.

## Why this and not "just sort and use libc bsearch"

You could keep one flat sorted array per set and call `bsearch` — simpler diff. But the ASCII fast path is the win: most characters in real input are ASCII, and the bitmap turns three linear scans into a single shift+AND. That's the case worth optimizing for.

## Out of scope for this fix

- `start_chars` / `end_chars` themselves are still consulted via `is_start_char` / `is_end_char`; no caller currently needs the raw arrays directly.
- Folding `is_inline_markup_start_char` and similar tiny `switch` classifiers into the bitmap scheme isn't worthwhile — they're already O(1) via the compiler's jump table.
